# 06 · Сохранения и облачная синхронизация

## Локальный сейв

### Формат
1. `PlayerSave` (Dictionary) → JSON-строка
2. JSON-строка → байты UTF-8
3. AES-256-CBC шифрование (ключ — производный от device-fingerprint + соль)
4. Запись в `user://save_0.bin`

### Триггеры сохранения
- `EventBus.save_dirty` (дебаунс 2 с)
- `NOTIFICATION_APPLICATION_PAUSED`
- `NOTIFICATION_WM_CLOSE_REQUEST`
- Перед `iap_completed` (страховка от потери покупки)
- Перед `prestige_triggered`

### Атомарность
Запись через временный файл + `rename` (атомарная операция файловой системы):
```
save_0.tmp → write → fsync → rename to save_0.bin
```

### Бэкап-слоты
- `save_0.bin` — текущий
- `save_1.bin` — предыдущий
- `save_2.bin` — позапрошлый
- При ошибке загрузки `save_0` → попытка `save_1`, затем `save_2`

## Облачный sync

### Базовая стратегия
- **Source of truth:** сервер
- При запуске клиент пушит локальный сейв на сервер с merge-стратегией
- При успешном пуше сервер возвращает merged state (если был конфликт)

### Конфликты
Конфликт = два устройства / две сессии с разными `last_seen_at` после общего предка.

Стратегии разрешения (в порядке выбора):
1. **Auto-merge** для аддитивных полей: ресурсы — max(), валюта — нет (см. ниже), достижения — union, уровни зданий — max
2. **Wins by score** для денег: state с большим `total_gold_earned` (накопительный счётчик, не изменяемый сбросом)
3. **Manual prompt** игроку в крайнем случае ("сохранение A: ур. 50, $1M / сохранение B: ур. 45, $2M")

### Anti-cheat при синхронизации
- Сервер сохраняет историю чекпоинтов и проверяет дельту между двумя сейвами
- Невозможные дельты (gold +∞, instant prestige) → флаг + rollback
- При флаге — клиент получает state предыдущего чекпоинта

### Endpoints
- `POST /save/push` body: `{save_blob, save_version, checksum}` → `{accepted: bool, merged_blob?}`
- `GET /save/pull` → `{save_blob, save_version, last_modified_at}`
- `POST /save/heartbeat` — раз в 60 с при активной игре, для anti-cheat

### Оффлайн-очередь
Если нет сети:
- Локальные изменения накапливаются
- При восстановлении сети — отправка пачкой
- На клиенте — индикатор «не синхронизировано» (не блокируем игру)

## Шифрование

### Ключ AES
- Не хардкодим в коде (легко достать обфускацией)
- Производный: `HKDF(device_id + obfuscated_salt + bundle_id)`
- При смене устройства — restore через cloud sync (там данные пишутся серверным ключом, не клиентским)

### Контекст
Цель шифрования — не «военная криптография», а:
- Защита от тривиального чит-редактирования файла на rooted/jailbroken
- Не помешать сохранению при offline

Серьёзная защита — серверная валидация (`10_backend.md`).

## Привязка к аккаунту

- **Гость:** device-id, локальная игра, опциональный cloud-сейв на anonymous slot
- **Login (Apple/Google/Email):** связывает прогресс с аккаунтом, синхронизация между устройствами

При первом логине из guest — клиент пушит локальный сейв с флагом `migration=true`. Сервер привязывает к аккаунту, если на нём ещё нет данных. Если есть — конфликт, prompt.

## Тестовые сценарии

- [ ] Сейв/загрузка идентичны (round-trip)
- [ ] Миграция с v1 → vN не теряет данных
- [ ] Коррапт `save_0.bin` → автоматический восстанов из `save_1.bin`
- [ ] Pause/resume не теряет последних 2 с прогресса
- [ ] Оффлайн → онлайн: очередь дочислится
- [ ] Два устройства, последовательная игра → корректный merge
- [ ] Два устройства, параллельная игра (≤5 мин) → conflict prompt
- [ ] Anti-cheat: ручное редактирование gold +1M → откат при следующем sync
