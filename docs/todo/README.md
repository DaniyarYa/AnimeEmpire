# todo/ — Конвенции

## Цель

Превратить геймдизайн (GDD) и архитектуру (`docs/architecture`) в конкретные исполнимые задачи, разбитые по фазам. Каждая фаза — самостоятельный этап с критериями выхода.

## Формат задачи

```
[ ] ID • Subject
    Acceptance: <как проверить, что готово>
    Estimate: S (≤ 1 день) | M (2-5 дней) | L (1-2 недели) | XL (> 2 недель → дробить)
    Depends on: ID или —
    Files: <главные файлы / пути>
    Notes: <опционально>
```

Префикс ID:
- `P0-*` Phase 0 (preprod)
- `P1-*` Phase 1 (vertical slice)
- `P2-*` Phase 2 (MVP)
- `P3-*` Phase 3 (soft-launch)
- `P4-*` Phase 4 (launch)
- `P5-*` Phase 5 (post-launch)
- `BL-*` Backlog (без фазы)

## Статусы

- `[ ]` pending — не начато
- `[/]` in-progress — в работе (с указанием исполнителя)
- `[x]` done — выполнено
- `[~]` blocked — заблокировано (указать чем)
- `[-]` cancelled — отменено

## Эпики

Внутри каждой фазы задачи группируются по эпикам (E1, E2, ...). Эпик — связанный набор задач, дающий единую ценность.

## Обновление

- TODO обновляется в каждом PR, который завершает / меняет задачу
- Раз в спринт (1-2 нед) — общий пересмотр приоритетов
- Закрытые задачи не удаляются (история), помечаются `[x]`

## Связь с tracker

Если команда использует Jira / Linear / GitHub Projects:
- Сами задачи дублируются в tracker для DOR/DOD/WIP-limits
- TODO в docs — source of truth для скоупа фаз и критериев выхода

## Критерии выхода фазы (DOD)

Каждая фаза имеет «Phase exit criteria» в начале файла. Переход к следующей фазе — только если все критерии выполнены.

## Срочные хотфиксы

Hotfix-задачи не привязаны к фазам, идут в `07_backlog.md` под секцией `Hotfix queue`.
